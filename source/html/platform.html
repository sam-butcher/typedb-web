In order to fully understand the platform, the Mindmaps Knowledge Representation (MKR) formalism must first be introduced. In Mindmaps everything is represented by a concept, and the figure to the left shows a simple example. Alice and Bob are concepts that represent individuals in a database, and the concept person represents the idea that they are people. In order to begin to add semantics to generic concepts in Mindmaps, and effectively classify them, concept types such as person can be defined. The arrows with the label ISA indicate that the users are the instances of the concept type person.

Another powerful construct is a relation that can assert information about concepts. It is useful, for example, to assert that a person is the director of a company and this is shown in the figure to the left. A specific company, Honesty Inc., has been defined here along with its concept type company. In addition, a relation type of directorship is defined giving semantics to the relation used to link the person and the company.

To increase the expressiveness of the knowledge representation a relation type is not enough information to specify the semantics of a relation. The roles in the relationship must also be defined. The figure to the left shows a relation type marriage with the role types husband and wife specified. For clarity the roles are reflected on the arrows pointing from the relation to the instances. This structure allows relationships with more than two roles to be expressed, which would not be possible using a single labelled arrow between Alice and Bob.

Mindmaps Knowledge Representation requires the identification of the concept types that play a role in a relationship. In the example shown in the figure to the left the arrows labelled plays role indicate that only a person can be a director, and only a company can be directed. This ensures that the semantics of the relationship are clearly defined, and that mistakes are not made when asserting the existence of relations between instances of specific types.

What has been outlined so far is an overview of the MKR formalism, based around concepts and the relations between them. A user of Mindmaps would first outline the concept, relation and role types and then specify which role types take part in a relation and which concept types can play a role. The classification of these concept types in the MKR formalism is depicted in the figure. The process of defining types results in an ontology that will govern the structure of the graph and expose the semantic power of data in Mindmaps. Once this has been done, data can be added as related instances to form a Semantic Graph, which serves as a Knowledge Base on top of which powerful applications can be built.

Core API

Mindmaps Core is an API with an object model that maps directly onto the Mindmaps Knowledge Representation (MKR) formalism. It enables the programmatic definition of an ontology, and interaction with data solely in terms of the MKR without having to consider the complexities of implementing it in a graph database. It abstracts away nodes and edges in your graph and allows your data to be structured in terms of concepts, the relationships between them, and the roles they play in these relationships.

An ontology can be thought of as a taxonomy of a particular domain of interest. An example of a minimal ontology for a Mindmaps graph is shown to the left. This ontology defines two full relationships, directorship and marriage, and specifies the roles in the relationships and the concept types that can play the roles. Factory

Mindmaps Factory is a semantic abstraction layer that produces a Mindmaps graph. It does so by structuring a graph database in such a way that it can be flexibly constrained and reasoned over. All that is required to persist a Mindmaps graph is a database that can represent property graphs. For this reason, Mindmaps supports graph databases that have implemented the Apache Tinkerpop API.

Mindmaps Factory provides the flexibility to use different graph databases in a semantic application. The persistence of a Mindmaps graph and its scalability, through Mindmaps Factory, are left to the graph database to handle. This means that an in-memory graph database can be used for small applications that run on a single machine. Conversely, a distributed and scalable graph database can be used to serve large volumes of traffic or to handle big data. This provides the freedom to choose the optimal database for a specific application.

Graql

Graql enables users to write queries against a Mindmaps graph leveraging the inherent semantics of the data. Concepts can be retrieved by specifying the patterns of types and relationships that identify them. Graql is declarative and therefore it handles the optimisation of the graph traversals needed to retrieve information.

A query for “all spouses of company directors” translates into asking for concepts that have the type person and are in marriage relationship with another person who is in a directorship relationship with a company. This is shown graphically and using Graql syntax in the figure to the left.

The select query form is used here to retrieve information from a Mindmaps graph. The types of the concepts and relations are specified using the isa syntax, and the relationships between the variables are expressed using brackets. There is no need to specify the roles in the above example because this information is implicit.

To extend simple retrieval operations, query modifiers can be specified that allow sorting, pagination, and distinct processing of the results. This allows queries such as “find all directors and order by their company”. The structure of the query is shown to the left with the Graql syntax. The ability to use the semantics of the data in a query makes the construction of very complex search patterns straightforward.

Graql also provides an interactive way to edit an ontology. Define concept types and relationships using the same syntax used to load data. In the example to the left the marriage relationship is defined and the structure in the graph shown. Here the Graql code uses the insert query form. This triggers the addition of data to Mindmaps, as opposed to a select query that returns data.

The additional query forms provided by Graql are: ask and delete. This provides all of the functionality needed to create ontologies, mutate data, and validate the existence of patterns - all of which are in terms of the Mindmaps Knowledge Representation.

Reasoner

Mindmaps Reasoner is our general purpose inference engine that provides a powerful mechanism for validating constraints and inferring new information in a Mindmaps graph. All constraints and inference rules are specified using Graql syntax and are automatically used to maintain data. Using Graql syntax to specify constraints and inference rules closely links the semantics of data to how it is constrained and reasoned over.

If we consider a system that maintains information about companies, a pragmatic constraint might be that all instances of company have a director. The first step is to identify all of the objects that are going to be constrained using a select query. The select query specifies a hypothesis, which is a pattern to be found in the graph. Once these patterns have been identified then an ASK query is used to specify the pattern that all results of the select query must match. These two queries constitute a constraint and this is shown in the graph.

The select query highlights the three instances of the concept type company. The ask query returns true for Honesty Inc. and Evilcorp, but false for TrustMe.com. This constraint would ensure that TrustMe.com is considered invalid until the required information is added. In this way, the information in a Mindmaps graph can be guaranteed to have a specified structure.

Graql

In order to infer new information from a Mindmaps graph, Mindmaps Reasoner needs to have a set of inference rules defined. Inference rules also consist of two Graql queries: a select and insert query. This select query is used to specify a pattern in the graph that the inference will act upon. For example, this pattern might identify suspicious relationships between people and companies. The insert query then specifies the action to perform if the hypothesised pattern is found to match any data. In this case, the action could be to flag the companies for further investigation. This is shown in the figure to the left.

Mindmaps Reasoner uses the select query to find that Honest Inc. and Evilcorp share an address and have related directors. It then uses the insert query to flag that the companies are suspicious. One of the features of Graql is that the type of the relationship between the people, “($u, $v)” - does not have to be specified, a relationship just needs to exist.

There are two ways of performing this inference, forward chaining and backward chaining, both of which are possible using Mindmaps Reasoner. The forward chaining operation of Reasoner responds in real-time to facts and rules added to the system, explicitly adding information to all matching patterns in the graph. Once a rule is added, all matching instances in the graph are found and the inference performed. Due to the nature of this approach, the inferred information may not be immediately available at query time while the system processes the rules.

The backward chaining operation is triggered when a query is executed. Mindmaps Reasoner translates the query, using the inference rules, in such a way that it finds information that implies the answer to the query. This is in addition to the information that directly satisfies the query. The backward chaining ensures that all inferred information is available at query time, even if it has not been explicitly added as a result of forward chaining.

Analytics

Mindmaps Analytics is currently under development and it is expected that the following tools will be included:

Graph Analytics - Discover the information contained in the structure of data. Evaluate the topology of a graph by investigating: small worldness, betweenness, centrality, degree distribution, modularity, mean path length, and other common algorithms. Discover how the concepts in a graph are connected and if they naturally form clusters or communities.

Easily specify a subgraph to analyse using faceted search and filtering operations. Working in terms of the Mindmaps Knowledge Representation easily define your subgraph by selecting a set of concept, relation and role types of interest.

Statistics - Compute aggregate measures across the concepts stored in Mindmaps. Find out how values related to concepts are distributed, regardless of the number of concepts. Count, sum, average, find the maximum, find the minimum and compute the mean of sets of concepts. The statistics component will be a toolbox of fundamental algorithms that scale with your data.

Usage Tracking - Monitor the usage of a Mindmaps graph. Determine the most visited concepts and paths traversed. Truly understand how users interact with the graph to optimise and personalise their experience. Extract information about what users are interested in and how these concepts are related.

Migration

In order to make migrating to Mindmaps hassle free, OWL and RDF Schema ontologies can be imported and automatically translated into a Mindmaps graph. This allows existing RDF triples to then be efficiently loaded into Mindmaps ready for use. If an ontology has not explicitly been defined, RDF triples can be loaded directly and an ontology will be automatically constructed. Custom tools are also provided for importing an SQL database into Mindmaps. Map an SQL schema into a Mindmaps ontology and load data without losing information.

Mindmaps Migration makes it simple to combine data from diverse sources into a single database. An ontology can be created that represents the individual data sources and the relationships between them. This in turn defines how the data is loaded to form a graph representing the union of data sets. Analysing and retrieving data from across sources then becomes a highly efficient operation due to the graph structure.

Mindmaps Migration provides batch loading to import large volumes of data efficiently. This entry point is particularly useful for building ETL tools to extract data from multiple sources and add to a Mindmaps graph.

Insight

In Mindmaps, knowledge is represented from the perspective of human understanding. Mindmaps Insight is a visual tool, that leverages the power of Mindmaps Knowledge Representation, for working with data. Navigate a Mindmaps graph interactively using faceted search and applying filters to refine what is displayed. Concepts can also be merged to represent aggregate information and see how these clusters are related.

Mindmaps Insight allows the results of Graql queries to be visualised as a graph. Even the specification of the queries can be done as a graph. Concept types and relationships can be combined in an intuitive way to define a query and refine it for the highest level of control. This makes it simple to create complex search patterns and edit them.

Within Mindmaps Insight a data model can be defined via an ontology. Define the set of concept types and how they are related using drag and drop tools. Constraints can then be applied directly to concept types and relation types for the greatest expressivity. This is also extended to allow the definition of constraint and inference rules using Graql queries and their interactive management.

Another aspect of Mindmaps Insight is the ability to monitor a Mindmaps cluster using dashboard visualisations and configure it via a web interface. This helps to identify pressure points on the infrastructure and track how a cluster is performing over time.
